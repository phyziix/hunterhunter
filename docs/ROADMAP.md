## Phase 2: 能力值系统 + 勋章系统独立 ✅ → v0.23 重新设计为「星点里程碑系统」

**阶段目标**：实现采集力、连接力、输出力三个能力值的计算和展示；将勋章系统从引擎代码中抽离，改为读取 `config.yaml` 的 `medals` 配置表。

**v0.23 更新**：废除独立能力值计算（采集力/输出力），改为累计星点档位制（500/1000/2000/3000 四档）。连接力保留独立计算，用于后台触发一次性星点奖励（≥5/+50、≥20/+100、≥50/+200）。废弃 `_calculate_hunt_power()`、`_calculate_output_power()`、`ability_changes` 日志（连接力变更日志保留）。新增 `cross_domain_bonus: 5`、`season_star_cap: 3000`、`season_soft_reset_ratio: 0.5` 配置。

**依赖**：Phase 1（数据结构迁移完成）

**预计工期**：3-4 天

### 任务清单

- [ ] **星点里程碑检查**
  - [ ] 在采集/发布后检查 `total_star` 是否跨越里程碑阈值（500/1000/2000/3000）
  - [ ] 到达新档位时触发对应增益效果（称号/倍率提升/输出下限/基金加成）
  - [ ] 通过勋章系统统一管理解锁触发

- [ ] **连接力计算（保留）**
  - [ ] 保留 `_calculate_link_power()` 函数
  - [ ] 公式：`密度 × 100 × log₂(节点数+1) × (跨界占比+0.1)`，范围 0.1~50
  - [ ] 在每次采集后更新连接力
  - [ ] 达到阈值（5/20/50）时触发一次性星点奖励

- [ ] **连接力变更日志（保留）**
  - [ ] 保留 `_update_ability_changes(ability="link_power", ...)` 函数（仅连接力）
  - [ ] 记录最近7天的变更（超过7天的自动清理）
  - [ ] 在前端能力面板展示变更摘要

- [ ] **跨界采集加成**
  - [ ] 采集时判断标签是否跨 ≥2 个领域
  - [ ] 跨界笔记额外 +5 星点（`cross_domain_bonus: 5`）

- [ ] **赛季星点上限**
  - [ ] 赛季内星点累计不超过 `season_star_cap: 3000`
  - [ ] 赛季结算时按 `season_soft_reset_ratio: 0.5` 保留星点

- [ ] **废弃代码清理**
  - [ ] 移除 `_calculate_hunt_power()` 函数
  - [ ] 移除 `_calculate_output_power()` 函数
  - [ ] 移除 `ability_changes` 中采集力/输出力的变更记录逻辑
  - [ ] 清理 `abilities` state 中的 `hunt_power` 和 `output_power` 字段（保留 `link_power`）

- [ ] **前端能力面板**
  - [ ] 改为展示累计星点 + 当前增益档位
  - [ ] 连接力变更日志（最近7条）
  - [ ] 展示解锁的称号和效果

- [x] **勋章系统独立（从引擎抽离）** ✅
  - [x] 实现 `check_medals()` 函数，读取 `config.yaml` 的 `medals` 配置表
  - [x] 原能力值解锁任务改为调用 `check_medals()`
  - [x] 实现 `get_all_medals()` 函数，返回所有勋章定义（含已获得/未获得状态）
  - [x] 实现 `get_medal_status(medal_id)` 函数，检查勋章是否已获得
  - [x] 勋章触发时机：
    - [x] 采集成功后检查 event 类型勋章（如 `daily_capture_count`）
    - [x] 能力值更新后检查 ability 类型勋章（如 `hunt_power >= 30`）
    - [x] 赛季结算时检查 season 类型勋章
  - [x] `once: true` 的勋章只能获得一次
  - [x] 前端勋章墙从 `get_all_medals()` 获取勋章列表
